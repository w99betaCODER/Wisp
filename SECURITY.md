# Security Policy

## Reporting a vulnerability

Wisp handles VPN credentials and payment state, so security reports are taken
seriously. **Please do not open a public issue for security problems.**

Instead, report privately via [GitHub Security Advisories]
(https://github.com/w99betaCODER/Wisp/security/advisories/new), or email the
maintainer. Include:

- a description of the issue and its impact,
- steps to reproduce or a proof of concept,
- affected version / commit.

You can expect an initial response within a few days.

## Scope notes

- The panel↔node link is secured with **mutual TLS** (TLS 1.3). Never expose a
  node agent without `WISP_TLS_*` configured — the plain-HTTP fallback is for
  local development only.
- The admin API is unauthenticated unless you set `WISP_API_TOKEN`. Always set
  it (and serve the dashboard over HTTPS) before exposing the panel; the token
  gates every `/api/*` route except the public branding endpoint and the
  HMAC-signed payment webhook.
- Payment webhooks are authenticated by an HMAC-SHA256 signature over the raw
  body (`WISP_WEBHOOK_SECRET`); the endpoint is disabled when the secret is unset.
- TLS keys generated by `wisp-certs` are written with `0600` permissions. Keep
  `ca.key` and `client.key` off the nodes.
